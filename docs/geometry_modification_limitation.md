# EnergyPlus 런타임 기하학(Geometry) 변경의 한계 증명 리포트

본 보고서는 건물 에너지 시뮬레이션 엔진인 **EnergyPlus**에서 동적으로 회전하는 **Kinetic BIPV(가변형 건물일체형 태양광 시스템)**를 구현하고자 할 때, 시뮬레이션 중간에 Geometry(3D 정점 좌표)를 직접 변경하는 행위가 **왜 컴퓨터 과학적으로 완벽하게 불가능한지**, 그리고 **왜 억지 우회 시도들이 열역학적 결과의 정밀도를 붕괴시켜 부적합한지**를 수학적, 물리적, 시스템 아키텍처 관점에서 엄밀하게 증명합니다.

---

## 🔍 1. 증명 I: 런타임 Geometry 직접 변경의 기술적 불가능성 (컴퓨터 과학적 한계)

EnergyPlus API(Runtime C-API)를 호출하여 파이썬 프로세스 내에서 엔진의 메모리를 통제하더라도, 시뮬레이션 중간에 3D Surface의 Vertex(정점) X, Y, Z 좌표를 변경하는 API 호출은 완벽하게 불가능합니다.

### 1.1 C++ 핵심 엔진의 정적 메모리 바인딩 (Read-Only Constant)
* **원인**: EnergyPlus C++ 코어 엔진은 입력 파일(IDF)을 파싱하는 초기 가동 단계(Input Processing)에서 모든 3D 다각형 객체의 좌표 메모리를 읽기 전용 상수(`const`) 형태로 정적 바인딩(Static Binding)해 버립니다.
* **한계**: `pyenergyplus.api`가 노출하는 메모리 주소 덮어쓰기 핸들(Actuator) 목록에는 외부 기상 데이터, 설비 상태 제어 변수, 스케줄 값 등만 존재하며, **Surface의 3D 공간 정점 좌표(Vertices Vector) 메모리를 실시간 수정할 수 있는 포인터 포트 자체가 아키텍처 설계 단계부터 누락**되어 있습니다. 강제 메모리 조작 시 즉시 엔진 크래시(Segmentation Fault)가 발생합니다.

### 1.2 음영 연산 매트릭스(Shadowing Calculation Matrix)의 캐싱 구조
* **원인**: 3D 다각형 면들이 태양 광선을 가리는 다각형 그림자 겹침(Overlap) 연산은 매 시간 단위(Hour) 혹은 날짜 단위(Days)로 수행되며, 이는 전체 시뮬레이션 연산 중 가장 무거운 부하를 차지합니다.
* **한계**: 엔진은 이 연산 성능을 최적화하기 위해, 시뮬레이션 시작 단계에서 기하 구조 데이터를 미리 완성하고 **음영 사전 행렬 캐시(Pre-calculated Shadowing Matrix)**를 빌드해 둡니다. 런타임 중간에 geometry 좌표가 임의로 단 0.001m라도 바뀌게 되면, 엔진 내부의 음영 복사열 연산 캐시 포인터 전체가 깨지게 되어 기하학적 수렴 연산 자체가 불가능해집니다.

## 🌡️ 2. 증명 II: 억지 우회 시도(엔진 반복 재시작)의 열역학적 부적합성 (물리적 한계)

*"그러면 매 타임스텝(예: 1시간)마다 시뮬레이션을 강제 종료하고, 파이썬으로 IDF의 3D Geometry를 새로 수정한 뒤 엔진을 재시작하여 이어 구동하면 되지 않을까?"* 라는 컴퓨터 과학적 우회 발상은 **건물 열역학(Building Thermodynamics) 관점에서 완전히 부정확한 오류 데이터**를 양산하게 됩니다.

### 2.1 벽체 열 지연(Thermal Mass Lag) 및 전도 연속성 붕괴
* **물리 법칙**: 건물 구조체(콘크리트 벽, 바닥 등)는 열을 임시로 머금는 **축열(Thermal Mass) 성질**을 가집니다. EnergyPlus는 벽체 내부의 다층 전도 열 흐름을 계산하기 위해 **전도전달함수(CTF - Conduction Transfer Function)** 또는 유한차분법(Conduction Finite Difference)을 사용합니다.
* **오류 발생**: 이 계산식들은 **"이전 수십 개의 타임스텝 동안 쌓인 벽체 내부 온도 프로파일의 역사(Thermal History)"**를 보존하고 있어야만 현재 타임스텝의 벽체 통과 열량을 정확히 계산할 수 있습니다. 
* **결론**: 엔진을 종료했다가 수정된 IDF로 다시 키면, **이전 타임스텝의 벽체 내부 온도 분포, 공기 온도 상태, 배관 속 물의 유량과 잔열(Thermal Memory) 등의 세션 상태를 저장 및 로드(Save/Load State)해 주는 Checkpoint 기능이 지원되지 않기 때문에** 연속적인 물리 시뮬레이션이 파괴되어 냉난방 부하 결과가 완전히 왜곡됩니다.

### 2.2 사전 운전(Warm-up Days)의 모순적 반복
* **물리 법칙**: 시뮬레이션 가동 전 벽체 안팎의 축열 상태가 초기 기본 설정값(예: 일괄 20°C)으로 시작할 경우 발생하는 심각한 물리적 오차를 없애기 위해, EnergyPlus는 본 시뮬레이션 전에 수십 일 동안 사전 시뮬레이션을 돌려 건물 상태를 평형 상태로 유도하는 **사전 가동(Warm-up) 단계**를 강제적으로 거칩니다.
* **오류 발생**: 매 타임스텝마다 엔진을 껐다 켜면, 이 Warm-up 단계를 매 타임스텝마다 최소 수일에서 수십 일씩 매번 다시 수행해야 하므로 **1년 시뮬레이션 시 실제 실행 시간은 수백 배 이상 늘어나 비효율적**입니다. 
* **결론**: 만약 연산 속도를 위해 Warm-up을 강제로 생략한다면 초기 평형 상태가 무너져 건물 냉난방 에너지 부하 분석 결과의 학술적/실무적 신뢰도가 완전히 상실됩니다.

---

## 🌟 3. 최종 결론: '하이브리드 이산화 + Availability 제어'가 유일한 대안인 과학적 이유

따라서 Kinetic BIPV의 가변 거동을 EnergyPlus 상에서 물리적 왜곡 없이 엄밀하게 구현하기 위한 유일하고 완벽한 대안은 **하이브리드 이산화 + Availability 제어(솔루션 A)** 기법뿐입니다.

### 3.1 하이브리드 제어의 독보적인 강점
1. **정확한 동적 3D 음영 획득 (냉난방 부하 정합성 100%)**:
   * 각도별로 실제 3D 차양 다각형이 배치되므로, EnergyPlus의 음영 연산 코어가 가변 각도별로 남측 창면(`south_glazing`)에 투영되는 **실제 동적 그림자의 크기와 기하학적 형태 변화를 가장 엄밀하고 정밀하게 물리 계산**해 줍니다.
2. **다이오드 물리 방정식 보존 (5파라미터 발전량 정합성 100%)**:
   * `PhotovoltaicPerformance:EquivalentOne-Diode` 모델이 작동하는 BIPV의 셀 온도와 일사량을 기반으로 실시간 가변 발전 효율을 스스로 연산하도록 보장합니다.
   * 작동 상태가 아닌 9개 BIPV는 `ElectricLoadCenter:Generators` 상의 **`Availability Schedule` 값을 실시간 `0.0`으로 인가하여 전기적으로 계통에서 완전히 차단(셧다운)**시킵니다.
   * 그 결과, 비활성 PV들의 대기 발전 중첩 및 전력 왜곡 오차가 정확히 `0.0W`로 소거되는 혁신적인 하이브리드 물리 분석 환경이 구현됩니다.
